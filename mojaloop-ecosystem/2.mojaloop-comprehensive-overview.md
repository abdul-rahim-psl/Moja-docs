deep on each service, with file:line citations against the checked-out source, and explicitly

oversimplify. It also draws on prior verified research already in this repo:
`docs/Kafka/kafka-topics-quote-transfer.md` and `docs/fsp-ingestion/*.md`, which independently

## 2. Account Lookup Service (ALS)

Source: `/home/abdul-rahim/mojaloop/account-lookup-service`.

### 2.1 Purpose & role

ALS answers "which DFSP owns this party identifier, in what currency?" *before* a quote/transfer
can be built. It does **not** decide this itself — it routes the question to a pluggable external
**Oracle** service and manages the registry of which Oracle handles which
`(partyIdType, currency)`. It also proxies participant/party registration (add/remove a
party↔FSP mapping) through the same Oracle mechanism.

### 2.2 API surface

Two separate Hapi servers: the main API server (`API_PORT`) and an Admin server (`ADMIN_PORT`)
(`src/lib/config.js`).

**Main API** (`src/api/routes.js`):
- `GET/POST/DELETE /participants/{Type}/{ID}[/{SubId}]` — register/query/deregister a
  party↔FSP mapping in the Oracle. `POST /participants` (no path params) is the batch variant
  (`src/domain/participants/participants.js:471`, `postParticipantsBatch`).
- `PUT /participants/{Type}/{ID}[/{SubId}]` and `.../error` — the callback *ALS itself receives*
  as a consumer, then relays onward.
- `GET /parties/{Type}/{ID}[/{SubId}]` — the actual discovery call a Payer DFSP makes.
- `PUT /parties/{Type}/{ID}[/{SubId}]` and `.../error` — callback carrying the resolved party.
- `DELETE /endpointcache` — force-flushes ALS's in-memory cache of central-ledger's participant
  endpoints (`src/api/endpointcache.js:45,58-59`).
- `GET /health` on both servers.

**Every mutating/lookup call returns HTTP 202 immediately** and does the real work in a
fire-and-forget async function whose only error handling is a `.catch()` for logging — confirmed
at `src/api/parties/{Type}/{ID}.js:79-83` and the four handlers in
`src/api/participants/{Type}/{ID}.js:78-83,121-127,171-175,213-217`. The actual result is
delivered later via ALS calling back out to a `PUT .../{Type}/{ID}` (or `.../error`) URL the
requester registered at onboarding (`src/models/participantEndpoint/facade.js:96,199`).

**Admin API** (`src/api/oracles.js`, `src/api/oracles/{ID}.js` → `src/domain/oracle/oracle.js`) —
CRUD for the Oracle registry, fully **synchronous** (no 202 pattern, since it's just a DB write):
- `GET /oracles[?type=&currency=]` (`oracle.js:106`)
- `POST /oracles` (`oracle.js:52`) — rejects if an active oracle already exists for that
  `(partyIdType, endpointType, currency)` tuple (lines 68-72).
- `PUT /oracles/{ID}` (`oracle.js:163`), `DELETE /oracles/{ID}` (`oracle.js:236`, soft-delete via
  `isActive=false`).

### 2.3 The Oracle routing model

`src/models/oracle/facade.js` is where routing decisions happen. `determineOracleEndpoint()`
(line 116) picks a cached-lookup strategy based on which of `currency`/`SubId` are present, then
queries the `oracleEndpoint` table for matching rows. **If more than one row matches, the one
with `isDefault: true` wins** — `isDefault` is a *tie-breaker among multiple applicable
registrations*, not a "nothing matched" fallback (e.g. `facade.js:257,295,333,373`). No match at
all throws an `ADD_PARTY_INFO_ERROR`.

The resolved Oracle is called with a **plain synchronous HTTP request/response** — `sendHttpRequest`
/`sendOracleGetRequest` (`facade.js:50-57,140-204`), `await`ed inline, no 202/callback semantics
between ALS and the Oracle. This is the opposite style from ALS's own DFSP-facing API (§2.2) —
worth being explicit about, since the two protocols are easy to conflate.

**ALS ships no built-in Oracle logic.** Confirmed by (a) `facade.js` containing only URL
templating + HTTP proxying, no identifier-resolution logic, and (b)
`ml-core-test-harness/docker-compose.yml:375-378` running a *separate* container,
`als-msisdn-oracle-svc`, that ALS's `oracleEndpoint` table points at as the reference MSISDN
oracle implementation.

### 2.4 Participant registry — misleadingly named

`src/domain/participants/participants.js` does **not** manage FSP/DFSP entities despite the name
— it manages party-identifier↔FSP *mapping records*, by proxying synchronous REST calls to
whatever Oracle is registered:
- `postParticipants` (line 363) — validates the requesting FSP exists via central-ledger
  (line 384), then `POST`s to the Oracle (line 387); on `201` sends a success `PUT` callback,
  else an error callback.
- `postParticipantsBatch` (line 471) — groups by `partyIdType`, issues one `oracleBatchRequest`
  per group (line 531), aggregates into one `PARTICIPANTS_BATCH_PUT` callback (line 553).
- `deleteParticipants` (line 595) — synchronous `DELETE` to the Oracle; optionally pre-validated
  by `validatePartyDeletion` if `Config.DELETE_PARTICIPANT_VALIDATION_ENABLED` (**default
  `false`**) to check the deleting FSP actually owns the record (`facade.js:206,232`) — i.e. this
  safety check is off by default.
- `getParticipantsByTypeAndID` (line 77) — validates requester exists (line 100), `GET`s the
  Oracle (line 108), `PUT`s `partyList[0].fspId` back to the requester (lines 109-132).

**No party↔FSP record is ever written to ALS's own database** — the write goes over HTTP to the
external Oracle, which owns its own storage. ALS's DB only ever stores routing metadata.

### 2.5 Proxy support (cross-scheme / hub-to-hub)

Per `docs/Proxy/Readme.md` and `docs/Proxy/Discovery.md` in the ALS repo: lets two independent
Mojoloop hub deployments ("schemes") interoperate by registering a "proxy participant" in each
scheme representing the other scheme's DFSPs, so they behave "as if they were a single scheme."
Stated assumptions: no shared identifiers across schemes, no cross-scheme limit checks, GET
`/transaction` resolved at the payee scheme, timeouts relaxed in non-payee schemes.

Flow (`src/domain/parties/services/GetPartiesService.js`): local Oracle lookup is tried first;
if it fails, ALS does a fan-out ("depth-first search" per the doc) `GET /parties/...` to every
registered proxy in parallel (`#sendOutProxyRequests`, line 260), tracking the pending fan-out
in Redis (`@mojaloop/inter-scheme-proxy-cache-lib`, keys `als:{sourceId}:{type}:{partyId}` and a
sibling `:expiresAt` key with ~40s TTL). When a proxy responds, `PutPartiesService.js:101`
back-fills the local Oracle with the new mapping so future lookups resolve locally without
repeating the fan-out. Entirely optional, gated by `PROXY_CACHE_CONFIG.enabled`
(`BasePartiesService.js:254`).

### 2.6 Database schema

ALS's own `migrations/` create **only**: `currency`, `endpointType`, `partyIdType`, and
`oracleEndpoint` (the registry table — FKs to the first three, plus `value` = oracle base URL,
`isDefault`, `isActive`). **No `participant`/`fsp` table exists anywhere in ALS's migrations** —
this is the strongest evidence that ALS does not own the participant registry (contrast §2.9).

Notable schema evolution: migration `09_oracleEndpoint-indexes.js` adds a unique constraint on
`(partyIdTypeId, endpointTypeId, currencyId)`; migration `10_oracleEndpoint-remove-constraints.js`
**drops it again** — meaning the "one active oracle per key" invariant is now enforced purely in
application code (`src/domain/oracle/oracle.js:68-72,181-185`), not the database.

### 2.7 Caching

Two layers of in-process Catbox-memory caching, plus one Redis-backed cache for a different
concern:
1. `src/models/oracle/oracleEndpointCached.js` — caches the DB rows describing which Oracle URL
   serves a given `partyIdType`/`currency`.
2. `src/models/oracle/oracleGetCached.js` — caches the **actual HTTP GET response from the
   external Oracle**, keyed by full request URL, so a repeat lookup for the same party ID doesn't
   re-hit the Oracle within the TTL.
3. Participant-existence and endpoint lookups against central-ledger are cached via
   `@mojaloop/central-services-shared`'s `Util.Endpoints`/`Util.Participants`, configured with
   separate TTLs (`CENTRAL_SHARED_ENDPOINT_CACHE_CONFIG: 180000ms`,
   `CENTRAL_SHARED_PARTICIPANT_CACHE_CONFIG: 3000ms`).

`src/lib/createDistLock.js` is unrelated to the above — a Redis-backed mutex
(`mutex:als-timeout-handler`) ensuring only one ALS replica runs the timeout-sweep cron at a time.

### 2.8 Timeout handling

`src/handlers/TimeoutHandler.js` runs a cron job (`CronJob.from`, gated by
`Config.HANDLERS_TIMEOUT_DISABLED`) that, under the distributed lock, calls
`timeoutInterschemePartiesLookups` (`src/domain/timeout/index.js:43-49`) — this sweeps
**inter-scheme proxy discovery requests that never got a response within the TTL window** (§2.5)
and sends a timeout callback to the original requester. This is specific to cross-scheme lookups,
not a generic "any unanswered lookup times out" mechanism — ordinary in-scheme Oracle calls rely
on the plain HTTP client timeout (`Config.HTTP_REQUEST_TIMEOUT_MS`, default 20s).

Note: a second timeout path, `timeoutProxyGetPartiesLookups`, exists in domain code
(`src/domain/timeout/index.js:51`) but its invocation is **commented out** in the registered
handler (`TimeoutHandler.js:54`) — a parked/half-finished feature.

### 2.9 Config & the real dependency on central-ledger

`SWITCH_ENDPOINT` (`src/lib/config.js:155`, resolves to `http://central-ledger:3001` in the
Docker default config) is used for **both**:
- Participant existence validation (`Util.Participants.getParticipant`)
- Participant endpoint resolution, to know where to send a result/error callback
  (`Util.Endpoints.getEndpoint`)

Both are called from `src/models/participantEndpoint/facade.js` on essentially every ALS request.
**ALS is therefore tightly runtime-coupled to central-ledger** despite having its own, separate
MySQL database — this contradicts any assumption that ALS is "fully independent."

### 2.10 Corrections to the public docs

- **"ALS implements... participant registry management"** — true only in the narrow sense of
  managing party↔FSP *mapping* records via Oracles. It does not own the FSP/participant entity
  registry (name, active flag, callback URLs) — that's central-ledger's, and ALS is a heavy
  synchronous client of it (§2.9).
- **"Oracles use synchronous responses"** — confirmed accurate (§2.3), but note this is opposite
  in style to ALS's *own* DFSP-facing API, which is fully async/202+callback (§2.2). Conflating
  the two is an easy mistake.

## 3. Quoting Service

Source: `/home/abdul-rahim/mojaloop/quoting-service`.

### 3.1 Purpose & the fee-calculation question — resolved

**Quoting Service does not calculate fees, commission, or any transfer amount.** It validates,
persists, and forwards. The payee DFSP's own backend computes fees/commission and returns them in
its `PUT /quotes/{id}` callback; quoting-service stores those numbers verbatim.

Evidence: `src/model/quotes.js:592-601` (`createQuoteResponse`, inside `handleQuoteUpdate`)
persists `transferAmount`, `payeeReceiveAmount`, `payeeFspFee`, `payeeFspCommission` straight off
the incoming payload — no arithmetic. A comment at line 631 (`// todo: create any additional
quoteParties e.g. for fees, commission etc...`) was never implemented. `forwardQuoteUpdate`
(lines 684-737) relays `originalQuoteResponse` — the untouched payee payload — to the payer. No
code path anywhere in `src/model/*` or `src/data/database.js` derives a fee amount.

This directly contradicts the `docs.mojaloop.io` phrasing ("calculation of possible fees and FSP
commission [performed by quoting-service]") — the repo's own README is more accurate: **"Both the
Payer and Payee FSP should calculate their part of the quote."** The docs describe the protocol
resource's *business function in the ecosystem*, not what code runs inside this microservice.

Quoting-service's actual computational role: FSPIOP header/participant validation,
duplicate/resend detection (SHA-256 hash comparison), DB persistence of the full quote lifecycle,
an optional `json-rules-engine` hook that can intercept/reroute/invalidate a quote before
forwarding (`src/model/rulesEngine.js`, rules loaded via `src/model/deps.js:33`), and HTTP
forwarding to the resolved FSP callback endpoint.

### 3.2 Request flow — `POST /quotes`

`src/api/quotes.js:59-128` builds a Kafka envelope (`dto.messageFromRequestDto`), produces to
`topic-quotes-post` (lines 114-117), returns 202 (line 120).

`QuotingHandler.handlePostQuotes` (`src/handlers/QuotingHandler.js:105-132`) → `QuotesModel.
handleQuoteRequest` (`src/model/quotes.js:176-408`):
1. `validateQuoteRequest` (lines 64-150) — checks the payer/payee DFSP has a position account for
   the currency, via a **direct DB query against central-ledger's own `participant`/
   `participantCurrency` tables** (`src/data/database.js:335-365`) — skipped if the DFSP resolves
   via the inter-scheme proxy cache instead. **No call to ALS anywhere in this path.**
2. `fetchParticipantsInfo` fetches payer/payee metadata via `GET ${SWITCH_ENDPOINT}/participants/
   ${dfspId}` (`src/lib/util.js:296-302`) — `SWITCH_ENDPOINT` defaults to
   `http://localhost:3001`, **central-ledger's admin-API port** (`central-ledger/config/
   default.json` → `PORT: 3001`) — confirming this is a call to central-ledger, cached in-memory.
3. Runs the optional rules engine (can terminate/reroute/reject).
4. Duplicate/resend check via SHA-256 hash against `quoteDuplicateCheck` (§3.4).
5. Resolves lookup-enum IDs, persists the quote in one Knex transaction: `createQuoteDuplicateCheck`
   → `createTransactionReference` → `createQuote` → `createPayerQuoteParty`/`createPayeeQuoteParty`
   → optional extensions/geoCode → commit (lines 279-351).
6. `forwardQuoteRequest` (lines 415-473) resolves the **payee's** callback endpoint by querying
   quoting-service's own copy of the `participantEndpoint`/`participant`/`endpointType` join
   (endpoint type `FSPIOP_CALLBACK_URL_QUOTES`, `src/data/database.js:675-694`) — **the same
   central-ledger-backed endpoint model ml-api-adapter uses for transfer callbacks** — then issues
   an HTTP `POST {endpoint}/quotes` with the original payload.

### 3.3 Response flow — `PUT /quotes/{ID}`

Same DTO→Kafka pattern, published to `topic-quotes-put` regardless of success/error — the
`isError` flag is computed but not used to pick a different topic (`src/api/quotes/{id}.js`).
`QuotingHandler.handlePutQuotes` (`src/handlers/QuotingHandler.js:134-158`) dispatches on
`payload.errorInformation` presence: `handleQuoteUpdate` vs `handleQuoteError`.

`handleQuoteUpdate` (`src/model/quotes.js:531-677`): rejects an `accept` header if present
(spec-mandated on callbacks), re-validates the responding DFSP has a position account, checks
duplicate/resend via `quoteResponseDuplicateCheck`, persists `createQuoteResponse` +
`createQuoteResponseIlpPacket` (raw ILP packet string, **no verification/regeneration of the ILP
condition happens here** — it's stored exactly as the payee sent it) in one transaction, then
`forwardQuoteUpdate` relays the untouched payee response to the payer's resolved callback URL.

### 3.4 Database schema

**Quoting-service has no `migrations/` directory of its own** — its `config/default.json`
connects (user `central_ledger`, schema `central_ledger`) to **central-ledger's MySQL schema**.
The quote tables are defined in `central-ledger/migrations/`, each explicitly commented "these
changes are required for the quoting-service and are not used by central-ledger":

| Table | Stores |
|---|---|
| `quote` | Core request: `quoteId` PK, transaction reference, expiration, initiator/scenario/amountType FKs, amount+currency |
| `quoteParty` | One row per payer/payee: identifier, `fspId`, resolved `participantId`, signed amount |
| `quoteResponse` | Payee's terms: `transferAmount`, `payeeReceiveAmount`, `payeeFspFeeAmount`, `payeeFspCommissionAmount`, `ilpCondition`, `isValid` |
| `quoteResponseIlpPacket` | Raw ILP packet blob |
| `quoteDuplicateCheck` / `quoteResponseDuplicateCheck` | SHA-256 hash per quoteId — resend/idempotency detection for request and response legs separately |
| `quoteError` | Persisted error callbacks |
| `quoteExtension` / `quotePartyIdInfoExtension` | Extension-list key/value pairs |
| `fxQuote`, `fxQuoteResponse`, `fxQuoteError`, `fxQuote*DuplicateCheck`, `fxQuoteConversionTerms(+Extension)`, `fxQuoteResponseConversionTerms(+Extension)`, `fxCharge` | FX-quote analogues, keyed by `conversionRequestId`/`conversionId` |

Quoting-service also **reads (but doesn't own/migrate)** central-ledger's core reference tables:
`participant`, `participantCurrency`, `participantEndpoint`, `endpointType`, and various
transaction-type/scenario/party-type lookup tables.

### 3.5 Bulk & FX quotes

- **Bulk quotes** (`src/model/bulkQuotes.js`) — same Kafka/dispatch pattern
  (`topic-bulkquotes-{post,put,get}`), same participant-position check, same endpoint-resolution
  and forwarding — but `handleBulkQuoteRequest` (lines 75-103) **does not persist anything to the
  database**; it validates and forwards immediately. Structurally thinner than single quotes.
- **FX quotes** (`src/model/fxQuotes.js`, ~870 lines) — mirrors single quotes closely, full
  transactional persistence on both POST and PUT, but validates against
  `LOCAL_ENUM.COUNTERPARTY_FSP` instead of PAYER/PAYEE (line 72) since FX quotes are between a
  DFSP and an FXP. Routed via the same top-level `src/api/quotes.js`/`quotes/{id}.js` files,
  branching on `isFX = util.isFxRequest(request.headers)` — not a separate API module.

### 3.6 Validation

- Participant/position-account existence (payer/payee or counterparty-FSP currency), skipped if
  proxied.
- `simpleRoutingMode`-specific validation of `partyIdInfo.fspId` against FSPIOP-Source/Destination
  headers (guards a routing-mode-specific spoofing case), `src/model/quotes.js:114-139`.
- Duplicate/resend detection via SHA-256 hash of the full body (`src/lib/util.js:218-222`): no
  prior record → proceed; same hash → idempotent resend, re-forward without re-persisting;
  different hash under the same ID → **hard error** (`MODIFIED_REQUEST`).
- `accept` header rejection on PUT callbacks.
- The `json-rules-engine` layer (`src/model/rulesEngine.js`) — the only actual "business rule"
  layer beyond structural checks; it can reject (`INVALID_QUOTE_REQUEST`) or reroute
  (`INTERCEPT_QUOTE`, rewriting `fspiop-destination`), but this is routing/rejection logic, not
  fee calculation.
- **No currency-conversion or amount-consistency check** on the response numbers
  (`transferAmount == payeeReceiveAmount + fees` is never verified) — the switch trusts the
  payee's numbers.
- No active expiration enforcement found at persistence time — `expirationDate` is stored, not
  checked against "now" in this code path.

### 3.7 Interactions with other services

- **No direct interaction with ALS anywhere** — no ALS URL in config, no code reference.
- **central-ledger, two ways**: (a) shared MySQL database (direct table reads/writes), (b) HTTP
  call to central-ledger's admin API (`config.switchEndpoint`) for participant metadata.
- **Payer/payee callback endpoints** are resolved from quoting-service's **own DB query** against
  the shared `participantEndpoint` table — not from a live call to ALS or central-ledger at
  request time.
- **Inter-scheme proxy cache** (`@mojaloop/inter-scheme-proxy-cache-lib`) resolves a representative
  proxy FSP for cross-scheme participants, independent of central-ledger.

### 3.8 Error handling

FSPIOP errors built via `@mojaloop/central-services-error-handling`. Any exception routes through
`handleException` → `sendErrorCallback` (`src/model/quotes.js:959-1103`), which resolves the
originating party's callback endpoint, builds headers (stripping/re-signing if switch-originated,
passing through if relaying a DFSP-originated error), and sends `PUT {endpoint}/quotes/{id}/error`
— JWS-signed if configured. Both success and error responses share `topic-quotes-put`,
discriminated only by `payload.errorInformation` presence — consistent with the earlier
Kafka-topics research in this repo.

## 4. Central Ledger

Source: `/home/abdul-rahim/mojaloop/central-ledger`.

### 4.1 Purpose & role

Per its own README (verbatim): "Brokering real-time messaging for funds clearing,"
"Maintaining net positions for a deferred net settlement," "Propagating scheme-level and
off-transfer fees." Central-ledger is a pure Kafka message processor for the transfer
prepare/fulfil lifecycle — it has no direct synchronous HTTP interaction with DFSPs at all; every
prepare/fulfil arrives as a Kafka message (produced by ml-api-adapter) and every result leaves as
a Kafka `NOTIFICATION` message (consumed by ml-api-adapter).

- **Net positions**: each DFSP+currency has one `participantPosition` row holding a running net
  `value`. Prepare reserves against it, fulfil/commit finalizes it, abort/timeout reverses it —
  positions accumulate rather than settle per-transfer (§4.5).
- **Off-transfer fees**: modeled via `ledgerAccountType` (`POSITION`, `SETTLEMENT`,
  `HUB_RECONCILIATION`, `HUB_MULTILATERAL_SETTLEMENT`, `INTERCHANGE_FEE`,
  `INTERCHANGE_FEE_SETTLEMENT`) and `ledgerEntryType` (`PRINCIPLE_VALUE`, `INTERCHANGE_FEE`,
  `HUB_FEE`) — a participant can carry a separate fee-position account alongside its principal
  position account, and a transfer can post to multiple ledger entries at once via
  `transferParticipant` rows (§4.4).

### 4.2 Kafka handlers — the full pipeline

Registration: `src/handlers/index.js` (CLI flags `--prepare/--position/--positionbatch/--get/
--fulfil/--timeout/--admin/--bulk*/--deferredSettlement/--grossSettlement/--rules`), wired in
`src/handlers/register.js:60-103`.

**Topics consumed**: `topic-transfer-prepare` (`prepare()`, `src/handlers/transfers/prepare.js:447`),
`topic-transfer-fulfil` (`fulfil()`, `src/handlers/transfers/handler.js:74`), `topic-transfer-get`
(`getTransfer()`, `handler.js:787`), `topic-position-prepare` (consumed by **either** the
non-batch handler, `src/handlers/positions/handler.js:282-296`, **or** the batch handler,
`src/handlers/positions/handlerBatch.js:226-251` — mutually exclusive deployment modes on the
same topic, distinguishing PREPARE/COMMIT/RESERVE/ABORT/REJECT/TIMEOUT_RESERVED/BULK_* by message
metadata, not topic name), `topic-admin-transfer` (manual reconciliation, `src/handlers/admin/
handler.js:173-186`), `topic-deferred-settlement-close` (`src/settlement/handlers/
deferredSettlement/handler.js:143-156`), plus bulk topics (`topic-bulk-{prepare,fulfil,processing,get}`).

FX position actions (`FX_PREPARE`, `FX_RESERVE`, `FX_TIMEOUT_RESERVED`, `FX_ABORT`) are
config-forced to `topic-transfer-position-batch` regardless of deployment mode
(`config/default.json` `KAFKA.EVENT_TYPE_ACTION_TOPIC_MAP.POSITION.*`) — **FX position changes
only work through the batch/bin pipeline, never the simple non-batch handler.** Conversely, per
comments in `prepare.js:405-408`, the batch handler doesn't support `bulk-prepare`. This is an
important, non-obvious operational constraint: batch vs non-batch is not just a performance
knob, it's a feature-support decision.

**Topics produced**: `topic-notification-event` (final result, consumed by ml-api-adapter),
`topic-transfer-position-{prepare,commit,abort}` (internal handoff from
transfer-handlers to position-handlers), `topic-bulk-processing` (bulk fan-out aggregation).

**The prepare → reserve → fulfil → commit → notify pipeline, precisely:**
1. `prepare()` decodes the message, resolves proxy/in-scheme obligations
   (`calculateProxyObligation`, line 147), runs a duplicate check against `transferDuplicateCheck`
   (line 220), validates the payload, persists the `transfer` row + `RECEIVED_PREPARE`
   `transferStateChange` (line 309).
2. Emits a `POSITION`-type message keyed on the **payer's** `participantCurrencyId`
   (`sendPositionPrepareMessage`, lines 382-424) to `topic-position-prepare`.
3. The position handler computes `liquidityCover` (negative of the payer's settlement-account
   position) and `availablePositionBasedOnPayerLimit` (NDC minus effective position); reserves
   (moves `participantPosition.value`, writes `RESERVED`) or rejects with
   `PAYER_FSP_INSUFFICIENT_LIQUIDITY`/`PAYER_LIMIT_ERROR` (`src/domain/position/prepare.js:150-233`).
4. On success, the payee is notified (RESERVED); when they fulfil, `fulfil()` validates FSPIOP
   headers against the transfer's `payerFsp`/`payeeFsp`, checks the fulfilment duplicate table,
   validates the ILP fulfilment against the stored condition, confirms state is `RESERVED` and
   not expired, then emits a `POSITION` COMMIT message keyed on the **payee's**
   `participantCurrencyId` (`handler.js:565-602`).
5. The position handler processes COMMIT: asserts state `RECEIVED_FULFIL`, moves the payee's
   position, writes `COMMITTED` (`positions/handler.js:174-198`).
6. A `NOTIFICATION` message is emitted (`positions/handler.js:195`) — this is what ml-api-adapter
   turns into the FSPIOP callback.

Abort/reject/timeout mirror this but call `changeParticipantPosition(..., isReversal=true, ...)`
to unwind the payer's original reservation (`positions/handler.js:199-221`).

**Important nuance on commit semantics**: on COMMIT, the payer's earlier reservation is *not*
separately reversed — the reservation **is** the final value (the payer already "paid" at
prepare time); fulfil only moves the payee side. Abort/timeout is what actually reverses the
payer's reservation, adding the amount back.

### 4.3 The position-batch handler — a real architectural fork, not just a perf knob

`src/handlers/positions/handlerBatch.js` consumes a Kafka **batch** (configurable `batchSize`),
groups messages into per-account "bins" keyed by `participantCurrencyId`, sub-groups by action,
opens **one DB transaction for the whole batch**, does one `SELECT ... FOR UPDATE` per account
(`getPositionsByAccountIdsForUpdate`) instead of one lock per message, and processes
PREPARE→FX_FULFIL→FX_TIMEOUT→FULFIL→ABORT→FX_ABORT→TIMEOUT_RESERVED→FX_PREPARE sub-processors
sequentially in memory per account via `src/domain/position/binProcessor.js:58` before a single
bulk write and one Kafka offset commit for the whole batch.

Note the processing order: **fulfils/timeouts/aborts of existing reservations are settled before
new prepares** within the same batch, so freed-up capacity from a completing transfer is available
to new prepares in the same pass. This turns N sequential row-locking transactions into one
transaction with N in-memory state passes — a genuine throughput optimization, at the cost of
much more complex accumulator-threading code (`accumulatedPositionValue`,
`accumulatedPositionReservedValue`, `accumulatedTransferStates` threaded through each sub-processor).

### 4.4 Position/limit/participant/fee model

- `participantPosition` — one row per `participantCurrencyId`: net `value` + `reservedValue`.
- `participantPositionChange` — append-only audit trail of every mutation.
- `participantLimit` — the only limit type actually enforced in the prepare path is
  **`NET_DEBIT_CAP` (NDC)** (`src/domain/position/binProcessor.js:158`).
- A `limitAlarms` early-warning list fires when `currentPosition > liquidityCover *
  thresholdAlarmPercentage` — a soft threshold below the hard NDC breach (line 289-292).
- `transferParticipant` — the double-entry-style ledger row per transfer, per role
  (PAYER_DFSP/PAYEE_DFSP), per `ledgerEntryTypeId` (PRINCIPLE_VALUE/INTERCHANGE_FEE/...) — this is
  what lets one transfer post to multiple ledger accounts (principal + fee) simultaneously.
- Each DFSP has one `participantCurrency` row per (currency × ledgerAccountType) — e.g. separate
  POSITION and SETTLEMENT accounts per currency, each with its own `participantPosition`.

### 4.5 Transfer lifecycle & DB schema

| Table | Purpose |
|---|---|
| `transfer` | Core row: `transferId` PK, amount, currency, ILP condition, expiration |
| `transferDuplicateCheck` | `transferId` + hash — idempotency guard against replayed prepares. `transfer` has an FK *to* this table, so every transfer must have a corresponding hash row |
| `transferParticipant` | Double-entry ledger row per transfer/role/entry-type |
| `transferStateChange` | Append-only FSM history: `RECEIVED_PREPARE → RESERVED → RECEIVED_FULFIL → COMMITTED` (or `ABORTED_REJECTED`/`ABORTED_ERROR`/`EXPIRED_RESERVED`) |
| `ilpPacket` | Raw ILP packet text per transfer — kept for fulfilment verification/dispute resolution |
| `transferFulfilment` | Fulfilment record — `ilpFulfilment`, `completedDate`, `isValid`, and **`settlementWindowId` already assigned at fulfil time** (central-ledger assigns settlement windows inline, not a downstream service) |
| `transferFulfilmentDuplicateCheck` | Same idempotency pattern as prepare, for the fulfil leg — a **separate** duplicate-check table per lifecycle phase |

Bulk mirrors all of the above one level up (`bulkTransfer`, `bulkTransferStateChange`, etc., plus
`bulkTransferAssociation` linking to individual transfer rows). FX mirrors the whole pattern under
separate `fxTransfer*` tables (§4.7).

### 4.6 Settlement — absorbed into central-ledger in this codebase

`settlementModel` (a named policy: `settlementGranularityId` NET/GROSS ×
`settlementInterchangeId` BILATERAL/MULTILATERAL × `settlementDelayId` IMMEDIATE/DEFERRED, per
currency or a default) is **configuration**, exposed via an admin HTTP API
(`src/api/settlementModels/*`). New participants are assigned the matching (or default)
settlement model when their currency account is created.

**Important, possibly-stale assumption to flag**: this checked-out version of central-ledger has
what looks like the **formerly-separate `central-settlement` service absorbed into itself**, under
`src/settlement/*` (routes, deferred/gross settlement handlers, settlement-window domain logic,
models). Evidence: `package.json` has no `@mojaloop/central-settlement` dependency, but its test
scripts explicitly include `test/settlement/**`; `src/api/routes.js:10` carries the comment *"This
can likely be removed, it was being consumed by central-settlements"* on the `ledgerAccountTypes`
route. So in this version, central-ledger does **both** real-time position bookkeeping **and**
settlement-window lifecycle management (open/close windows, compute net settlement obligations)
— a mental model of "central-ledger does positions, central-settlement does settlement" would be
outdated here. The `ml-core-test-harness` docker-compose still runs a separate
`central-settlement` container image, though — **this is a real, unresolved tension worth
confirming against whichever production version is actually deployed**, since it's possible the
two services coexist with overlapping-but-different responsibilities, or that the standalone
image is legacy/unused in this particular stack.

The deferred-settlement handler (`src/settlement/handlers/deferredSettlement/handler.js:60-133`)
consumes a `CLOSE` action and closes a settlement window (with retries), which is the trigger that
aggregates `transferFulfilment` rows assigned to that window into net settlement obligations per
participant.

### 4.7 FX support — the `cyril.js` matching engine

`fxTransfer` (keyed by `commitRequestId`, not `transferId`) is a structurally parallel table to
`transfer`, carrying `sourceAmount`/`sourceCurrency` and `targetAmount`/`targetCurrency`, plus an
optional `determiningTransferId` linking it to the payment transfer it's converting funds for.

`src/domain/fx/cyril.js` (~500 lines) is the reconciliation engine. Core mechanism:
- `fxWatchList` rows (keyed by `commitRequestId` + `determiningTransferId` + `fxTransferTypeId`:
  `PAYER_CONVERSION` or `PAYEE_CONVERSION`) record which FX conversion belongs to which underlying
  transfer, **regardless of which message arrives first** — `checkIfDeterminingTransferExistsFor
  FxTransferMessage` (line 81) figures out if the transfer or the FX-prepare came first, and
  therefore whether this is a payer-side conversion (reserve against the requesting DFSP in
  source currency) or a payee-side conversion (reserve against the receiving FXP in target
  currency).
- On transfer fulfil, `processFulfilMessage` (lines 347-494) looks up all watch-list entries for
  that `determiningTransferId` and builds a chain of position obligations across up to **three or
  more parties**: the original payer, the original payee, and one or two FXPs (a sending-side and
  a receiving-side conversion can both apply to the same transfer). All resulting position moves
  are applied as one sequence inside a single bin-processor pass
  (`src/domain/position/fulfil.js:95-152` walks `cyrilResult.positionChanges`, re-emitting itself
  as a "followup" message until every leg is processed).
- `fx-prepare.js`/`fx-fulfil.js`/`fx-timeout-reserved.js` under `src/domain/position/` are the
  bin-processor equivalents of the plain transfer handlers, reachable **only** via the batch
  pipeline (§4.2/§4.3).

In short: an FX transfer generalizes the normal 2-party ledger entry into an N-party chain of
position moves, correlated across two independently-ordered async message streams via the
watch-list. Pre-existing sequence diagrams exist in `central-ledger/documentation/
fx-implementation/` and `documentation/sequence-diagrams/` if a visual walkthrough is needed.

### 4.8 Bulk transfers — a two-database feature

Bulk uses a **hybrid storage model**: SQL tracks state/duplicate-hashes (`bulkTransfer`,
`bulkTransferStateChange`, `bulkTransferDuplicateCheck`, `bulkTransferAssociation`), but the
**actual bulk payload** (the list of individual transfers) is stored in **MongoDB** via
`@mojaloop/object-store-lib` (`src/handlers/bulk/prepare/handler.js:44,55-58`) — a materially
different persistence path from individual transfers, which live entirely in the relational
schema. Position-side, bulk actions reuse the exact same position/bin-processor pipeline as
individual transfers (`BULK_PREPARE`/`BULK_COMMIT`/etc. branches throughout
`src/handlers/positions/handler.js`), just routed to a `BULK_PROCESSING` event type instead of
`NOTIFICATION` so results funnel through bulk-specific aggregation before the FSP is notified.

### 4.9 Proxy cache (cross-scheme, ledger side)

`src/lib/proxyCache.js` wraps `@mojaloop/inter-scheme-proxy-cache-lib` (same library ALS uses,
§2.5, but a separate Redis-backed instance/concern here):
- `getFSPProxy` — if a `dfspId` isn't a known in-scheme participant, looks up which proxy
  represents it.
- `checkSameCreditorDebtorProxy` — detects when payer and payee are both represented by the
  *same* proxy (both foreign to this scheme) — in that case no position adjustment happens in
  this scheme at all (`prepare.js:356-366`, `messageKey='0'`).
- `getProxyParticipantAccountDetails` — resolves which `participantCurrencyId` to actually post a
  position change against (real participant vs. representing proxy) — used pervasively by
  `cyril.js` for FX position-chain construction.

### 4.10 Admin API

`src/api/participants/handler.js` + `routes.js`:
- `POST/GET/PUT /participants[/{name}]` — create/list/get/update a DFSP.
- `POST/GET /participants/{name}/endpoints` — register/query FSPIOP callback URLs (e.g.
  `FSPIOP_CALLBACK_URL_TRANSFER_POST`) — **this is the exact table ml-api-adapter and
  quoting-service both read from** to resolve where to send callbacks
  (`src/models/participant/facade.js:267`, `getEndpoint(participantId, endpointType)`).
- `POST /participants/{name}/initialPositionAndLimits`, `GET/PUT /participants/{name}/limits` —
  set/get NDC and initial position.
- `GET /participants/{name}/positions`, account CRUD for adding e.g. an INTERCHANGE_FEE account.
- Manual reconciliation: `PUT /participants/{name}/accounts/{id}/transfers/{transferId}` and the
  Kafka-driven `RECORD_FUNDS_IN`/`RECORD_FUNDS_OUT_*` flow (`src/handlers/admin/handler.js:51-97`,
  topic `topic-admin-transfer`) — lets an operator post ledger entries directly, bypassing the
  normal prepare/fulfil flow.
- `createHubAccount` — creates the Hub's own reconciliation/multilateral-settlement accounts,
  gated by `Config.HUB_ID`/`HUB_ACCOUNTS`.

### 4.11 Config

`RC.HUB_PARTICIPANT.*` (`HUB_ID`/`HUB_NAME`/`HUB_ACCOUNTS`) identifies the Hub's own ledger
participant. `PROXY_CACHE_CONFIG` and `KAFKA_CONFIG.EVENT_TYPE_ACTION_TOPIC_MAP` were covered
above. **Central-ledger does not call out to other services by URL from static config** — unlike
`ml-api-adapter`, there's no `ENDPOINT_SOURCE_URL`-equivalent here, because central-ledger *is*
the source of truth for the `participantEndpoint` table everyone else reads from.
`MONGODB_*` config is used only by the bulk-transfer object store and can be disabled.

## 5. ML API Adapter

Source: `/home/abdul-rahim/mojaloop/ml-api-adapter`. This service was already covered in depth by
prior verified research in this repo (`docs/Kafka/kafka-topics-quote-transfer.md`,
`docs/fsp-ingestion/*.md`) — summarized here with additions from this pass.

### 5.1 Role

The FSPIOP-facing HTTP↔Kafka gateway for the transfer resource family. It is a **producer-only**
participant for `POST/PUT/GET /transfers` (and `/fxTransfers`) — it never consumes the topics it
produces to; central-ledger is the consumer. It separately runs a **notification consumer**
(`src/handlers/notification/index.js`) on `topic-notification-event`, which is where
central-ledger's processing result comes back in, gets turned into an FSPIOP `PUT`/`PATCH`/error
callback, and is sent to the FSP's registered endpoint.

### 5.2 Kafka topic map (previously verified live against a running stack)

| Topic | Produced by | Consumed by |
|---|---|---|
| `topic-transfer-prepare` | `POST /transfers` → `src/domain/transfer/index.js` `prepare()` | central-ledger |
| `topic-transfer-fulfil` | `PUT /transfers/{ID}` `fulfil()`/`transferError()` (commit/reserve/reject/abort all ride this topic, distinguished by an `action` field in the message) | central-ledger |
| `topic-transfer-get` | `GET /transfers/{ID}` `getTransferById()` | central-ledger |
| `topic-notification-event` | central-ledger | `src/handlers/notification/index.js` — resolves destination FSP endpoint, sends outbound callback |

Confirmed live (per the prior research in this repo): single partition/single replica per topic
in the local stack; `topic-notification-event` fires **once per state transition**, not once per
request — 37 messages observed on that topic for 8 requests each on the prepare/fulfil topics in
one golden-path run, and it also fires for settlement-related actions, not just quote/transfer
state changes. A production consumer of this topic needs to filter/dedupe by
`metadata.event.action` rather than treat every message as a distinct business event.

Payload encoding differs by resource: quote payloads are plain JSON; **transfer payloads are
base64-encoded** inside a `data:application/vnd...;base64,...` URI — the one structural difference
a generic Kafka consumer needs to branch on between the two topic families.

### 5.3 Security & interoperability layers (this pass's additions)

- **JWS signing/validation** — config keys `JWS_SIGN`/`JWS_VALIDATE` (default both `false`),
  `JWS_SIGNING_KEY_PATH`, `JWS_VERIFICATION_KEYS_DIRECTORY` — off by default, opt-in per
  deployment.
- **ISO 20022 support** — `src/interface/api-swagger-iso20022-transfers.yaml` and
  `fspiop-rest-v2.0-ISO20022_transfers.yaml` exist alongside the plain FSPIOP swagger, and
  `src/lib/payloadCache/PayloadCache.js` is a Redis-backed cache (`RedisCache` from
  `central-services-shared`, key prefix `iso_payload`, default TTL 300s) — this exists
  specifically to hold onto an original ISO 20022 payload by `requestId` across the
  request/response cycle (needed because FSPIOP↔ISO20022 translation requires the original
  message available when building a response, unlike plain FSPIOP where the payload doesn't need
  round-tripping). Confirms `ml-api-adapter` supports being deployed with `API_TYPE: iso20022` in
  addition to plain `fspiop`.
- **Distributed tracing** — `@mojaloop/event-sdk` is a dependency; the prior Kafka-topic research
  confirmed a `trace` block (span/trace IDs, W3C tracestate) rides on every Kafka message across
  both quote and transfer topics, letting a consumer correlate messages across topics/services by
  `traceId`.
- **Config-modifier pattern** — the local test harness starts ml-api-adapter via
  `config-modifier/run.js`, which layers `config-modifier/configs/ml-api-adapter.js` on top of
  the static `config/default.json` to set hostname, `ENDPOINT_SOURCE_URL` (→ central-ledger),
  Kafka broker addresses, and Redis-backed proxy-cache settings at container start — i.e. most
  environment-specific wiring happens through this modifier script, not raw env vars alone.

### 5.4 Shared library stack (applies across all four services)

`package.json` confirms the common dependency set that gives every core Mojoloop Node service
consistent behavior: `@mojaloop/central-services-shared` (Kafka wrapper, Enums, HTTP util,
participant/endpoint cache helpers), `-error-handling` (FSPIOP error model), `-stream` (Kafka
consumer/producer primitives), `event-sdk` (tracing), `sdk-standard-components` (JWS, ILP
packet handling). This is why the topic-naming convention, error envelope shape, and callback-URL
resolution pattern look identical across ALS, quoting-service, central-ledger, and
ml-api-adapter — they're built from the same libraries, not independently reinvented per service.

## 6. Cross-Cutting Concerns

### 6.1 The 202-then-callback pattern is universal

Every FSPIOP ingress endpoint across all three HTTP-facing services (ALS, quoting-service,
ml-api-adapter) returns an immediate `202 Accepted` and delivers the actual result via a
separately-initiated `PUT`/`PATCH` call to a URL the FSP registered at onboarding. This is not a
per-service quirk — it's the foundational async design of the whole FSPIOP protocol, and every
service implements it the same way (fire-and-forget internal function + later outbound HTTP call
to a cached, DB-resolved endpoint).

### 6.2 The participant-endpoint model is central-ledger's, shared by everyone

There is exactly one authoritative source of "where do I send a callback for participant X, of
type Y (`FSPIOP_CALLBACK_URL_QUOTES`, `..._TRANSFER_POST`, `..._PARTIES_PUT`, etc.)" — the
`participant`/`participantEndpoint`/`endpointType` tables owned and admin-API-managed by
central-ledger. ALS, quoting-service, and ml-api-adapter all resolve callback destinations against
this same data (ALS and ml-api-adapter via HTTP + `central-services-shared`'s caching helpers,
quoting-service via a direct DB query since it shares central-ledger's schema). This is arguably
the single most important shared architectural fact in the whole ecosystem: **there is one
participant registry, and it lives in central-ledger**, not in ALS despite ALS's name suggesting
otherwise.

### 6.3 Idempotency is enforced per-phase, not per-resource

Every async, two-phase (or multi-phase) protocol resource in this ecosystem has a **separate**
duplicate-check table per phase, keyed by a SHA-256 hash of the request/response body:
`transferDuplicateCheck` (prepare) + `transferFulfilmentDuplicateCheck` (fulfil) in central-ledger;
`quoteDuplicateCheck` (request) + `quoteResponseDuplicateCheck` (response) in quoting-service
(sharing central-ledger's DB); FX and bulk variants of both. The pattern is uniform: same hash →
treat as a resend and replay the prior response; different hash under the same ID → hard reject
as a modified/duplicate request. This is a deliberate, repeated design decision, not incidental.

### 6.4 Two families of "proxy" support solve the same cross-scheme problem independently

Both ALS (`docs/Proxy/*`, `GetPartiesService.js`, `proxyCache`) and central-ledger
(`src/lib/proxyCache.js`, used in `prepare.js` and `cyril.js`) implement inter-scheme/cross-hub
routing using the same underlying library
(`@mojaloop/inter-scheme-proxy-cache-lib`) but as two **separate** Redis-backed caches solving
two different sub-problems: ALS resolves *discovery* across schemes (does a foreign scheme own
this party?), central-ledger resolves *settlement obligations* across schemes (which account,
real or proxy, does a position change post against?). They are not the same cache or code path,
despite the shared library and shared vocabulary — worth not conflating when reasoning about a
cross-scheme transaction end to end.

### 6.5 The "Hub" is a first-class ledger participant, not just an abstraction

Central-ledger's config seeds a `Hub` participant (`HUB_ID`/`HUB_NAME`/`HUB_ACCOUNTS`) with its
own reconciliation and multilateral-settlement accounts. Scheme-level fees (interchange, hub
fees) are posted to this participant's accounts via the same `transferParticipant`/
`ledgerEntryType` mechanism used for ordinary DFSP-to-DFSP transfers — the switch operator is
modeled as just another participant in its own ledger, not a special-cased side system.

### 6.6 Caching is pervasive and multi-layered, for the same underlying reason everywhere

Every service caches the same two kinds of thing for the same reason (avoid a DB join or
external HTTP round-trip on every single transaction): (a) routing/endpoint metadata (ALS's
`oracleEndpointCached`, everyone's participant-endpoint cache via `central-services-shared`), and
(b) in some cases the actual external response (ALS's `oracleGetCached`, ml-api-adapter's
ISO20022 `PayloadCache`). None of these caches are the primary source of truth — central-ledger's
MySQL tables are — they exist purely for hot-path latency at transaction volume.
