# P2P Flow â€” Code Follow

A code-level walkthrough of the standard P2P payment flow described in
[`3.standard p2p payment flow.md`](3.standard%20p2p%20payment%20flow.md). This doc skips the
narrative/sequence-diagram framing and instead follows the flow **hop by hop through the actual
source**, across the four core services: `account-lookup-service`, `quoting-service`,
`ml-api-adapter`, `central-ledger`. MLA/PPA/Tazama are intentionally out of scope here â€” see doc 3
for those.

Legend: đźź  HTTP hop Â· đź”µ Kafka publish Â· đźź˘ Kafka consume

---

## Stage 1 â€” Party Discovery (`account-lookup-service`)

Purely synchronous HTTP underneath â€” **no Kafka involved at all** in this stage (confirmed in doc 3).

### 1.1 đźź  Payer FSP asks "who owns this ID?"

`GET /parties/{Type}/{ID}` lands on ALS. The controller fires the domain lookup **without
awaiting it**, then returns 202 immediately â€” the real answer comes later via callback.

`account-lookup-service/src/api/parties/{Type}/{ID}.js:75-84`
```js
await validateSourceFspHeader(headers)

// Here we call an async function- but as we send an immediate sync response, _all_ errors
// _must_ be handled by getPartiesByTypeAndID.
parties.getPartiesByTypeAndID(headers, params, method, query, span, proxyCache).catch(err => {
  request.server.log(['error'], `ERROR - getPartiesByTypeAndID: ${LibUtil.getStackOrInspect(err)}`)
})
histTimerEnd({ success: true })
return h.response().code(202)
```

### 1.2 đźź  ALS resolves the owner via the Oracle

The detached call above eventually reaches `BasePartiesService.sendOracleDiscoveryRequest()`,
which issues a plain synchronous HTTP GET to the external Oracle registry.

`account-lookup-service/src/domain/parties/services/BasePartiesService.js:173-181`
```js
async sendOracleDiscoveryRequest () {
  this.stepInProgress('sendOracleDiscoveryRequest')
  const { headers, params, query } = this.inputs

  const response = await this.deps.oracle.oracleRequest(headers, RestMethods.GET, params, query, undefined)
  this.log.verbose('oracle discovery raw response:', { response })

  let { partyList } = response?.data || {}
  ...
```

### 1.3 đźź  ALS calls the requester back directly

Once the Oracle answers, ALS delivers the result by calling the **payer FSP's own registered
callback URL** â€” a direct outbound HTTP `PUT`, not a Kafka message.

`account-lookup-service/src/domain/parties/services/PutPartiesService.js:87-99`
```js
async sendSuccessCallback () {
  const { headers, params, dataUri } = this.inputs
  const sendTo = this.state.requester
  this.stepInProgress('sendSuccessCallback')
  const payload = PutPartiesService.decodeDataUriPayload(dataUri)
  const callbackEndpointType = this.deps.partiesUtils.putPartyCbType(params.SubId)
  const options = this.deps.partiesUtils.partiesRequestOptionsDto(params)

  await this.deps.participant.sendRequest(
    headers, sendTo, callbackEndpointType, RestMethods.PUT, payload, options
  )
  this.log.verbose('sendSuccessCallback is sent', { sendTo, payload })
}
```

Flow so far: `api/parties/{Type}/{ID}.js` (202) â†’ `domain/parties/getPartiesByTypeAndID.js` â†’
`GetPartiesService` (Oracle) â†’ `PutPartiesService` (callback to payer). Both `sendRequest` calls
bottom out in `@mojaloop/central-services-shared`'s shared HTTP client â€” external to this repo.

## Stage 2 â€” Quote Agreement (`quoting-service`)

Same HTTP-202-then-Kafka pattern used everywhere else in the switch.

### 2.1 đźź đź”µ `POST /quotes`

The quote-request handler builds the Kafka message, produces it to `topic-quotes-post`, **then**
responds 202.

`quoting-service/src/api/quotes.js:114-120` (trimmed)
```js
const producerConfig = isFX ? kafkaConfig.PRODUCER.FX_QUOTE.POST : kafkaConfig.PRODUCER.QUOTE.POST
const topicConfig = dto.topicConfigDto({ topicName: producerConfig.topic })
step = 'produceMessage-2'
await Producer.produceMessage(message, topicConfig, producerConfig.config)

histTimerEnd({ success: true })
return h.response().code(Http.ReturnCodes.ACCEPTED.CODE)
```

Consumed by `QuotingHandler.handlePostQuotes` (group `group-quotes-handler-post`), which â€” outside
Kafka â€” forwards the request over HTTP to the payee FSP.

### 2.2 đźź đź”µ `PUT /quotes/{ID}` â€” payee's answer comes back

The payee FSP responds with agreed terms (`ilpPacket`, `condition`, fees). This handler publishes
to `topic-quotes-put` â€” the **same topic used for the error path** too (flagged via
`payload.errorInformation`, not a separate topic).

`quoting-service/src/api/quotes/{id}.js:198-202` (trimmed)
```js
const producerConfig = isFX ? kafkaConfig.PRODUCER.FX_QUOTE.PUT : kafkaConfig.PRODUCER.QUOTE.PUT
const topicConfig = dto.topicConfigDto({ topicName: producerConfig.topic })
await Producer.produceMessage(message, topicConfig, producerConfig.config)

histTimerEnd({ success: true })
return h.response().code(Http.ReturnCodes.OK.CODE)
```

Both handlers build the Kafka envelope via `messageFromRequestDto` in `quoting-service/src/lib/dto.js`
â€” see doc [`kafka-topics-quote-transfer.md`](../Kafka/kafka-topics-quote-transfer.md) for the
message shape. Quoting-service is both producer and consumer of its own topics â€” no other service
is involved in Stage 2.

## Stage 3 â€” Transfer Execution (`ml-api-adapter` + `central-ledger`)

This is where two services split responsibility: **ml-api-adapter is HTTP-in/Kafka-out only** â€” it
never touches transfer business logic. **central-ledger owns everything** â€” position checks,
reservation, settlement, and the outbound notification.

### 3.1 đźź đź”µ `POST /transfers` â€” ml-api-adapter produces, doesn't process

`ml-api-adapter/src/domain/transfer/index.js:58-68` (trimmed)
```js
const prepare = async (headers, dataUri, payload, span, context = {}, isIsoMode) => {
  const logPrefix = `domain::${payload.transferId ? 'transfer' : 'fxTransfer'}::prepare`
  ...
  let messageProtocol = dto.prepareMessageDto({ headers, dataUri, payload, logPrefix, context, isIsoMode })
  messageProtocol = await span.injectContextToMessage(messageProtocol)
  const { topicConfig, kafkaConfig } = dto.producerConfigDto(Action.TRANSFER, Action.PREPARE, logPrefix)

  await Kafka.Producer.produceMessage(messageProtocol, topicConfig, kafkaConfig)
  return true
}
```
Publishes to `topic-transfer-prepare`. The HTTP controller (`ml-api-adapter/src/api/transfers/handler.js`)
returns 202 right after this resolves.

### 3.2 đźź˘ central-ledger consumes prepare, checks the payer's position

`central-ledger/src/handlers/transfers/prepare.js:447-465` is the consumer entry point
(registered in `handler.js:976-990`). After payload/participant validation it hands off to the
**position** topic rather than settling inline:

`central-ledger/src/handlers/transfers/prepare.js:545-548`
```js
logger.info(Util.breadcrumb(location, `positionTopic1--${actionLetter}7`))
const success = await sendPositionPrepareMessage({
  isFx, action, params, determiningTransferCheckResult, proxyObligation
})
```

The **position handler** (a separate consumer) does the actual liquidity/limit check and moves the
transfer to `RESERVED`:

`central-ledger/src/models/position/facade.js:178-194` (trimmed)
```js
if (new MLNumber(availablePositionBasedOnLiquidityCover).toNumber() < transferAmount.toNumber()) {
  transferState.transferStateId = Enum.Transfers.TransferInternalState.ABORTED_REJECTED
  transferState.reason = ErrorHandler.Enums.FSPIOPErrorCodes.PAYER_FSP_INSUFFICIENT_LIQUIDITY.message
  ...
} else if (new MLNumber(availablePositionBasedOnPayerLimit).toNumber() < transferAmount.toNumber()) {
  transferState.transferStateId = Enum.Transfers.TransferInternalState.ABORTED_REJECTED
  transferState.reason = ErrorHandler.Enums.FSPIOPErrorCodes.PAYER_LIMIT_ERROR.message
  ...
} else {
  transferState.transferStateId = Enum.Transfers.TransferState.RESERVED
  sumReserved = new MLNumber(sumReserved).add(transferAmount).toFixed(Config.AMOUNT.SCALE)
}
```

This RESERVED state transition is what ultimately reaches the payee (fund reservation notice),
via the notification path described in 3.5 below.

### 3.3 đźź đź”µ `PUT /transfers/{ID}` â€” payee fulfils

Once the payee FSP releases funds/goods, it calls back with the fulfilment. Same producer-only
pattern as prepare:

`ml-api-adapter/src/domain/transfer/index.js:88-98` (trimmed)
```js
const fulfil = async (headers, dataUri, payload, params, span, context = {}, isIsoMode) => {
  ...
  let messageProtocol = dto.fulfilMessageDto({ headers, dataUri, payload, params, logPrefix, context, isIsoMode })
  messageProtocol = await span.injectContextToMessage(messageProtocol)
  const { topicConfig, kafkaConfig } = dto.producerConfigDto(Action.TRANSFER, Action.FULFIL, logPrefix)

  await Kafka.Producer.produceMessage(messageProtocol, topicConfig, kafkaConfig)
  return true
}
```
Publishes to `topic-transfer-fulfil` (aborts/rejects reuse this same topic with a different
`action` value inside the message â€” no separate error topic).

### 3.4 đźź˘ central-ledger consumes fulfil, commits the payee's position

Entry point: `central-ledger/src/handlers/transfers/handler.js:74-88` (`fulfil`, registered at
`handler.js:1000-1013`). On `COMMIT`/`RESERVE` it re-publishes to the position topic, which the
position handler picks up to do the actual settlement write:

`central-ledger/src/handlers/positions/handler.js:174-197` (trimmed)
```js

  [Enum.Events.Event.Action.COMMIT, Enum.Events.Event.Action.RESERVE, Enum.Events.Event.Action.BULK_COMMIT].includes(action)) {
  const transferInfo = await TransferService.getTransferInfoToChangePosition(
    transferId, Enum.Accounts.TransferParticipantRoleType.PAYEE_DFSP, Enum.Accounts.LedgerEntryType.PRINCIPLE_VALUE)
  ...

    transferId: transferInfo.transferId,
    transferStateId: Enum.Transfers.TransferState.COMMITTED
  }
  await PositionService.changeParticipantPosition(participantCurrency.participantCurrencyId, isReversal, transferInfo.amount, transferStateChange)
```

The DB write itself â€” the actual ledger entry:

`central-ledger/src/models/position/facade.js:285-295` (trimmed)
```js

latestPosition = latestPosition.toFixed(Config.AMOUNT.SCALE)
await knex('participantPosition').transacting(trx).where({ participantCurrencyId }).update({
  value: latestPosition,
  changedDate: transactionTimestamp
})
await knex('transferStateChange').transacting(trx).insert(transferStateChange)
```

This is the actual moment of settlement in the flow â€” the row in `participantPosition` is where
the money "moves."

### 3.5 đź”µ central-ledger produces the notification event

There's no dedicated `notification` handler file in central-ledger â€” both the prepare and position
handlers set `eventDetail.functionality = NOTIFICATION` and hand off to the shared `Kafka.proceed`
helper, which publishes to `topic-notification-event`.

`central-ledger/src/handlers/positions/handler.js:142-147`
```js
const eventDetail = { action }
if (![...bulk actions].includes(action)) {
  eventDetail.functionality = Enum.Events.Event.Type.NOTIFICATION
} else {
  eventDetail.functionality = Enum.Events.Event.Type.BULK_PROCESSING
}
```
```js
// handler.js:195
await Kafka.proceed(Config.KAFKA_CONFIG, params, { consumerCommit, eventDetail, hubName: Config.HUB_NAME })
```
`Kafka.proceed` itself lives in the external `@mojaloop/central-services-shared` package, not in
this repo â€” but this is the exact call site that triggers it. **This topic fires once per state
transition**, not once per request â€” 37 messages were observed on one 8-request golden-path run
(see doc `kafka-topics-quote-transfer.md`).

### 3.6 đźź˘đźź  ml-api-adapter consumes the notification, calls the FSP back

This is the final hop â€” where the flow leaves Kafka and becomes an HTTP callback the payer/payee
FSP actually receives.

Consumer entry point: `ml-api-adapter/src/handlers/notification/index.js:107-142` (`startConsumer`)
wires `consumeMessage` (line 133), which calls `processMessage` per Kafka message (line 184).
`processMessage` (lines 244-1072) decodes the event, resolves the destination FSP's registered
callback endpoint, and sends the outbound HTTP request. Example â€” the `COMMIT`/`RESERVE` success
branch that becomes the fulfilment `PUT` the payer sees:

`ml-api-adapter/src/handlers/notification/index.js:465-469, 497-508` (trimmed)
```js
if ([Action.COMMIT, Action.RESERVE, Action.FX_COMMIT, Action.FX_RESERVE].includes(action) && isSuccess) {
  const callbackURLTo = await getEndpointFn(destination, REQUEST_TYPE.PUT)
  const endpointTemplate = getEndpointTemplate(REQUEST_TYPE.PUT)
  ...
  headers = createCallbackHeaders({ dfspId: destination, transferId: id, headers: content.headers, httpMethod: PUT, endpointTemplate }, false)
  response = await sendHttpRequest({
    url: callbackURLTo,
    headers,
    source,
    destination,
    method: PUT,
    payload,
    span
  })
}
```

`sendHttpRequest` wraps `Callback.sendRequest` (`ml-api-adapter/src/handlers/notification/index.js:359-370`),
itself built on the shared `Util.Request.sendRequest` client â€” same external shared lib used by
ALS in Stage 1.

thing to receive a message in this flow, closing the loop it opened in Stage 1.
